# Event consequences (Phase 6)

**Layer:** 2–3 bridge (downtime / world events)  
**Status:** Implemented  
**Modules:** [`shared/eventConsequence.ts`](../../shared/eventConsequence.ts), [`backend/src/lib/eventConsequenceService.ts`](../../backend/src/lib/eventConsequenceService.ts)

## Principle

**Events describe narrative reality; consequences are explicit operator actions.**

World events (`CalendarEvent` / chronology `world_event`) record what happened. Consequences are GM-authored, manually applied effects that ratify canon — not autonomous simulation.

## Storage

- Metadata on event-lore wiki pages: `event-{calendarEventId}`
- Key: `eventConsequences` (`event-consequence-v1`)
- Lazy lore pages set `eventLoreAutoGenerated: true` with a minimal markdown stub
- **Not** stored on `CalendarEvent.conditions` (temporal gating only)

## Consequence kinds (v1)

| Kind | Layer | Behavior |
|------|-------|----------|
| `quest_hook` | 2 | `discover_quest` or `open_thread` via `payload.mode` |
| `alter_location` | 3 | Append-only `downtimeAlterations` on location wiki |
| `route_change` | 3 | DRAFT trade-route overlay; narrative `severity` / `reason` mapped internally |
| `haven_threat` | 3 | `applyHavenThreatPatch()` — activity log + threat entry |

## Identity & application

- Stable `id` from `computeEventConsequenceId(kind, payload, targets)` — no receipt table
- `application.state`: `pending` | `complete` | `partial` | `blocked`
- Only `complete` rows are skipped on re-apply
- `previewOnly` apply runs dispatch without persisting substrate or metadata changes

## API

| Method | Path | Access |
|--------|------|--------|
| GET | `/c/:slug/calendar-events/:eventId/consequences` | Chronology manager |
| PUT | `/c/:slug/calendar-events/:eventId/consequences` | Chronology manager |

| POST | `...?previewOnly=true` | Dry-run preview |

## UI — World Impact (GM-facing)

The authoring surface uses **World Impact** framing. Internal enums (`quest_hook`, `discover_quest`, etc.) are not shown to GMs.

### Label mapping

| Internal kind | GM label |
|---------------|----------|
| `quest_hook` | Opportunity appears |
| `alter_location` | Location changes |
| `route_change` | Travel disruption |
| `haven_threat` | Haven threatened |

| Application state | GM label |
|-------------------|----------|
| `pending` | Not yet applied |
| `complete` | Applied |
| `partial` | Needs review |
| `blocked` | Could not apply |

### Authoring behavior

- **Event lore page** (`/c/:slug/event-:id`) — `EventConsequencesEditor` (GM): template cards, page pickers, lore-based suggestions, preview, apply
- **Quest hook mode** — `discover_quest` vs `open_thread` is inferred from the linked page (`QUEST` template → discover quest; otherwise open thread). No mode dropdown.
- **Preview panel** — narrative bullet summaries; count headline only when multiple impacts or mixed outcomes
- **Downtime overview** — actionable `event_consequence` feed cards with the same narrative copy (priority above creative drift)

Presentation helpers: [`shared/eventConsequencePresentation.ts`](../../shared/eventConsequencePresentation.ts)
